## 2. Fastest Deployable Product (FDP)

The **Fastest Deployable Product (FDP)** is a **Daily Operations Log & Financial Ledger** designed for a single user (the School Principal, Teacher, or Accountant) to manage invoicing, take daily attendance, and collect fees.

```
┌──────────────────────────────────────────────────────────────┐
│            FASTEST DEPLOYABLE PRODUCT BOUNDARY               │
├──────────────────────────────┬───────────────────────────────┤
│       INCLUDED (P0)          │         EXCLUDED (P2)         │
├──────────────────────────────┼───────────────────────────────┤
│ - Admin/Accountant Login     │ - Dedicated Teacher Portal    │
│ - Student Profile Registry   │ - Dedicated Parent App        │
│ - Staff Simple CRUD          │ - Homework & Diary Feeds     │
│ - Excel Import (Roster Upload)│ - Online Payment Gateway      │
│ - Daily Attendance Register  │ - SMS API Integrations        │
│ - Term Bill Batch Invoicing  │ - Automated Year Rollover     │
│ - FIFO Cashier Payments Desk │ - PDF Render Engine (QuestPDF)│
│ - Printer-Friendly Receipt   │ - Concessions Portal Grid     │
└──────────────────────────────┴───────────────────────────────┘
```

* **Included Modules:** Authentication, Tenant Metadata, Academic Configuration, Classes/Sections setup, Staff Directory (Simple CRUD), Student directory, Excel Bulk Import, Daily Attendance Grid, Fee Line Items Configuration, Invoice Directory & Batch Invoicing, Cashier Desk (FIFO allocations, manual receipt generation), and Defaulter Reports.
* **Excluded Modules:** Concessions portal workflows, Exams, Grading spreadsheets, Homework, Notification message brokers, Transfer Certificates, Parent Login portals, and Teacher Dashboards.
* **Risks:**
  1. *Manual Alert Fatigue:* School must message parents manually since SMS APIs are excluded.
  2. *Single Point of Failure:* If the school accountant's laptop/internet fails, they must revert to paper bills.
* **Tradeoffs:** We trade automated notifications and parent portal logins for **speed-to-market**. By providing the school owner with daily operational utility (attendance) and accurate cash collection sheets, we solve their biggest immediate business problems (parent engagement and receivables management) while writing 50% less code.

## 3. First Customer Roadmap (6-Week Delivery)

This roadmap focuses on building the system progressively over three 2-week phases.

```
┌─────────────────────────────────────────────────────────────┐
│ PHASE A: The Core Registry & Staff Directory (Weeks 1 - 2)   │
│ - Tenants, Users, Staff CRUD, Classes, and Fees Config      │
└───────────────────────────────┬─────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────┐
│ PHASE B: Roster, Attendance & Billing (Weeks 3 - 4)          │
│ - Student Database, Excel Import, Attendance Grid, Invoicing │
└───────────────────────────────┬─────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────┐
│ PHASE C: Collection, Reporting & Deployment (Weeks 5 - 6)    │
│ - Cashier Desk, Defaulter Reports, WhatsApp Web URL Hack    │
└─────────────────────────────────────────────────────────────┘
```

### Phase A: Core Registry & Staff Directory (Weeks 1 - 2)
* **Goal:** Configure academic boundaries, classes, simple staff profiles, and basic billing line items.
* **Database Tables Required:** `Tenants`, `TenantSettings`, `Users`, `Roles`, `UserRoles`, `AcademicYears`, `Classes`, `Sections`, `ClassSections`, `Staff`, `FeeGroups`, `FeeLineItems`.
* **APIs Required:**
  * `POST /api/auth/login`
  * `GET /api/settings` & `PUT /api/settings`
  * `POST /api/academic-years` & `GET /api/academic-years`
  * `POST /api/classes` & `GET /api/classes`
  * `POST /api/class-sections` & `GET /api/class-sections`
  * `POST /api/staff` & `GET /api/staff` & `GET /api/staff/{id}` & `PUT /api/staff/{id}`
  * `POST /api/fees/line-items` & `GET /api/fees/line-items`
* **Screens Required:**
  * `LoginScreen` (Minimal styled card, credentials inputs).
  * `AdminDashboard` (KPI stats cards: Total Invoiced, Collected, Balance).
  * `StaffDirectory` (Simple list grid of employees).
  * `ClassSectionConfigurator` (Grid listing grade mappings with `ClassTeacherId` dropdown).
  * `FeeStructureConfigurator` (Setting rates per grade).

### Phase B: Roster, Attendance, & Invoicing (Weeks 3 - 4)
* **Goal:** Build the student registry, bulk upload school data, configure daily attendance registers, and run term billing.
* **Database Tables Required:** `Students`, `StudentClassHistory`, `Guardians`, `StudentGuardians`, `DailyAttendance`, `StudentInvoices`, `StudentInvoiceDetails`.
* **APIs Required:**
  * `POST /api/students` & `GET /api/students/{id}` & `PUT /api/students/{id}`
  * `POST /api/students/import` (Fast Excel bulk parser)
  * `GET /api/students/search`
  * `GET /api/attendance/roster` & `POST /api/attendance/submit` & `POST /api/attendance/unlock`
  * `POST /api/invoices/generate-batch`
  * `GET /api/invoices/{id}` & `GET /api/invoices`
* **Screens Required:**
  * `StudentSearchDirectory` (Data table with filter dropdowns and upload drawer).
  * `StudentProfileForm` (Standard details card with parent phone field).
  * `DailyAttendanceSheet` (Roster list grid with large, mobile-friendly `P`, `A`, `L`, `T` checkboxes).
  * `InvoiceDirectory` (Invoice index with "Generate Batch" button).

### Phase C: Collection, Reporting, & Deployment (Weeks 5 - 6)
* **Goal:** Create payment collection workstation, generate defaulter reports, build browser-based receipts printing, and deploy services to target servers.
* **Database Tables Required:** `FeeReceipts`, `FeeReceiptAllocations`.
* **APIs Required:**
  * `POST /api/receipts/collect` (FIFO algorithm)
  * `POST /api/receipts/{id}/cancel` (Reversals)
  * `GET /api/reports/fee-defaulters`
* **Screens Required:**
  * `CashierPaymentDesk` (Dues summary sheet, payment amount inputs, FIFO allocation preview).
  * `ReceiptHistoryDesk` (Void triggers and list of collections).
  * `SchoolReportsDesk` (Print-friendly HTML table of outstanding student balances).
  * `TenantSettingsDashboard` (Identity parameters).
* **GTM WhatsApp Hack Implementation:** On the `SchoolReportsDesk` view, configure a button next to each defaulter that uses `window.open("https://wa.me/{parentPhone}?text={prefilledMessage}")` to compose a payment reminder in WhatsApp Web, bypassing the need for integrated SMS APIs.

## 4. 80/20 Analysis (Impact vs. Effort)

```
                       80/20 VALUE GRID
     ┌───────────────────────────────────┬───────────────────────────────────┐
     │           HIGH EFFORT             │            LOW EFFORT             │
     │         LOW VALUE (DEFER)         │       HIGH VALUE (BUILD FIRST)    │
  H  ├───────────────────────────────────┼───────────────────────────────────┤
  I  │                                   │                                   │
  G  │  * Online PG Gateways             │  * Excel Bulk Roster Upload       │
  H  │  * CBSE QuestPDF Engines          │  * Batch Invoice Generator        │
     │  * Multi-step Year Rollovers      │  * FIFO Payments Cashier Desk     │
  V  │                                   │  * Daily Attendance Grid          │
  A  ├───────────────────────────────────┼───────────────────────────────────┤
  L  │                                   │                                   │
  U  │  * Live Chat Modules              │  * Browser Print Dues Reports     │
  E  │  * Profile Photo Compression      │  * WhatsApp Web Quick-Alerts      │
     │                                   │                                   │
     └───────────────────────────────────┴───────────────────────────────────┘
                       HIGH EFFORT                 LOW EFFORT
```

### 20% of Features Delivering 80% of Business Value:
1. **Excel Bulk Roster Upload:** Instantly imports thousands of student records, cutting manual entry work from days to minutes.
2. **Daily Attendance Grid:** Provides teachers and staff with a daily utility workflow that establishes system use habits.
3. **Batch Invoicing Engine:** Generates terminal bills for all classes with a single click, instantly setting up the receivables ledger.
4. **FIFO Cashier Desk:** Accountant can collect cash or record UPI bank transfers, print receipts on local paper, and track accurate student balances in real-time.
5. **HTML/Browser Defaulter Reports:** Generates clean tables that can be printed or shared as PDFs using the browser's native print menu (`Ctrl+P`).

### Features with High Development Effort but Low Immediate Customer Value:
1. **QuestPDF Engine:** Custom C# code styling report cards/receipts takes weeks to build and align, whereas simple HTML tables with `@media print` CSS configurations render just as well for MVP billing reports.
2. **Razorpay/PG Webhooks:** Requires handling network dropouts, webhook retries, transaction validation, and onboarding delays, while 98% of target local schools prefer cash or direct bank transfers for the first few users.
3. **Automated Rollover Wizard:** A multi-step wizard requires complex validation code. The rollover event occurs only once a year, meaning a manual SQL script run by the developer at year-end provides the same value for the MVP.

## 5. Features to Remove/Defer From Version 1

| Feature | Action | Business & Technical Justification |
| :--- | :---: | :--- |
| **Homework** | **Defer** | Defer to Phase 2. Teachers and parents already communicate homework via WhatsApp. Building a custom feed adds zero onboarding incentive. |
| **Exams** | **Defer** | Defer to Phase 2. Schools do not grade students during the first 3 months of the term. Deferring this frees up weeks of development time. |
| **Report Cards** | **Defer** | Defer to Phase 2. Without exams, there are no report cards. Deferring eliminates the need for complex template builders. |
| **Notifications** | **Defer** | Defer to Phase 2. Integrated alerts require DLT registration, template approvals, and SMS API integrations. |
| **Transfer Certificates** | **Remove** | Remove from MVP. TCs are issued infrequently. The school principal can write them in Word templates. |
| **Year Rollover** | **Remove** | Remove from MVP. Rollovers only occur at the end of the academic year. Building this on day 1 is unnecessary. |
| **QuestPDF Reports** | **Remove** | Remove from MVP. QuestPDF is excellent but requires code-based rendering. Standard HTML browser-based printing is faster to build and sufficient. |
| **WhatsApp Integration** | **Defer** | Defer to Phase 2. Integrated API notifications are costly and complex. Replaced with dynamic WhatsApp client links. |
| **Razorpay Integration** | **Defer** | Defer to Phase 2. Schools collect fees via cash/cheque/UPI direct. Removing PG integration avoids merchant validation bottlenecks. |
| **Concessions UI Board** | **Defer** | Defer to W5-6. Minor manual discount adjustments can be typed straight into the specific student invoice details rather than designing a structured concession approval lifecycle board for MVP. |

## 8. Go-To-Market (GTM) 45-Day Revenue Run

To secure a paying customer within 45 days, you must focus on speed-to-value. 

### 1. What to Build (P0):
* The daily attendance sheet, which acts as a daily dashboard for the staff.
* The batch billing engine and the cashier payment desk. These are the tools that allow the school to bill their students and record cash collections.
* The Excel student importer, which allows you to onboard the pilot school's roster in 10 minutes during the initial demo.
* A simple Staff list, ensuring that attendance mark logs and class teacher indicators match DB integrity.

### 2. What to Postpone (P2):
* The parent app and online gateways. The school can log UPI reference numbers manually.
* The automatic rollover logic. Perform any rollovers manually in the database if the pilot school finishes their academic year early.
* Carrier SMS APIs.

### 3. What to Fake/Hack Manually:
* **The "SMS Gateway":** Do not configure integrated SMS gateways. On the student directory and defaulter reports screens, configure buttons that generate a `https://wa.me/` dynamic URL on click. When clicked, this opens the accountant's WhatsApp Web with a pre-filled, personalized reminder template (e.g., *"Dear Rajesh, Tuition fee balance of ₹5,200 is outstanding for Amit. Please pay at the front desk."*). This is free, requires zero compliance setups, and parents are more likely to read a direct WhatsApp message than an SMS.
* **The "Online Payment Portal":** Do not build a payment gateway integration. Place the school's bank account details and a UPI QR code image on the printed invoice. Parents pay using GooglePay/PhonePe to the school's bank account, and the school accountant manually records the payment transaction in the cashier desk, logging the transaction reference ID.
* **Database Rollover:** If a pilot school needs to roll over to a new year during the pilot phase, run the SQL promotion script manually in the database rather than writing a UI rollover engine.
* **Backup Auditing:** Rely on standard database backups and a simple transaction history log rather than building a complex JSON audit logs viewer screen.
