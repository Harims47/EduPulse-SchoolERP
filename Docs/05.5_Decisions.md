## 1. Student Lifecycle Management

### 1.1. Decision Area: Student Status Lifecycle
Defining a complete, immutable student status lifecycle to maintain transactional auditing, registration records, and historical student logs.

```
       [ Applied ] ──► [ Admitted ] ──► [ Active ] ────┬──► [ Promoted ] ──► [ Active (New Class) ]
                                         │             │
                                         ▼             ├──► [ Repeating ] ──► [ Active (Same Class) ]
                                    [ Transferred ]    │
                                    [ Dropped ] ◄──────┴──► [ Graduated ]
                                         │
                                         ▼
                                    [ Archived ]
```

### 1.2. Recommended Approach
Every student profile contains a `Status` field bound to a strict database lookup schema:
* **Applied:** Registration completed by parent; fee payments pending. No class or roll number assigned.
* **Admitted:** Initial fee cleared; class and section assigned. Awaiting active term start.
* **Active:** Attending classes in the current academic year. Appears in daily attendance grids.
* **Promoted:** Successfully completed the academic year and eligible for the next class. Set at year-end.
* **Repeating:** Academic thresholds not met. Maintained in the same class level for the next academic year.
* **Transferred:** Issued an official Transfer Certificate (TC). Profile locked.
* **Graduated:** Completed Grade 12 (the final school standard). Profile locked.
* **Dropped:** Voluntarily left without a TC or long-term absent without tuition fee settlements.
* **Archived:** Historical record retained for audit trails. Non-modifiable.

### 1.3. Business Justification
A complete lifecycle ensures compliance with CBSE audit rules, prevents unpaid or unregistered students from appearing in active classroom registries, and ensures fee structures are applied only to billable profiles.

### 1.4. Database Impact
* Enforced enum constraints (`VARCHAR(20)`) check on `Students.Status`.
* Unique index creation: `AdmissionNo` is unique across all statuses; `RollNo` is unique *only* within a specific `ClassSectionId` when status is `Active` or `Repeating`.

### 1.5. API Impact
* API checks block attendance inputs for any student whose status is not `Active` or `Repeating`.
* Profile editing endpoints lock record properties when state changes to `Transferred`, `Graduated`, or `Archived`.

### 1.6. Future Scalability Impact
Historical databases can easily archive students without corrupting historical billing summaries or transcript data.

## 2. Academic Year Promotion & Rollover Workflow

### 2.1. Decision Area: Year-End Rollover & Data Preservation
Determining how to preserve past records (attendance, test scores, invoices, class logs) during promotions.

### 2.2. Recommended Approach
Implement an **Active Year Cohort mapping pattern** using a junction table (`StudentClassHistory`) to decouple class memberships from the main student record.
* **Historical Data Retention:** Attendance, marks, and invoices are linked to an `AcademicYearId` and the student's unique profile keys. They are never overwritten or deleted.
* **Prior Dues Rollover:** Unpaid fee balances are calculated at rollover, closed out in the previous term ledger, and written to the new term ledger as an "Opening Balance Due" line item.
* **Rollover Execution Steps:**
  1. Verify all final exams are marked as "Published".
  2. Batch create new `StudentClassHistory` entries for the upcoming year based on student status (`Promoted` -> Class+1, `Repeating` -> Class).
  3. Change active student statuses to `Active` under the new year.
  4. Toggle the Global Tenant configuration `ActiveAcademicYearId` to the new year.

### 2.3. Business Justification
Schools must keep transcripts, financial audits, and attendance registers for up to 10 years to comply with board regulations. Decoupling class mappings ensures past reports render correctly even after a student changes sections.

### 2.4. Database Impact
* Creation of `StudentClassHistory` table:
  * `StudentId` (FK)
  * `ClassSectionId` (FK)
  * `AcademicYearId` (FK)
  * `RollNo` (Integer)
* The `Students` primary table does *not* contain class or roll numbers directly; it references current active cohorts.

### 2.5. API Impact
* The API connection builder resolves `SESSION_CONTEXT('AcademicYearId')` to filter active rosters automatically.
* Legacy APIs require an explicit `@YearId` parameter to read historical datasets.

### 2.6. Future Scalability Impact
Prevents query degradation over time. Historical records can be partitioned by `AcademicYearId` in SQL Server as database volume grows.

## 3. Parent / Guardian Relationship Model

### 3.1. Decision Area: Parent-Child Linkage
Model a structure that supports siblings, multiple guardians, and emergency contact details.

### 3.2. Recommended Approach
Adopt a **Many-to-Many Sibling-Capable Guardian Junction** design.

```
  [ Student Table ]                      [ Guardian Table ]
  - StudentId (PK)                       - GuardianId (PK)
  - Name                                 - Phone / Email (User Auth ID)
         │                                      │
         └─────────────► [ Junction ] ◄─────────┘
                         - StudentId (FK)
                         - GuardianId (FK)
                         - RelationshipType (Father/Mother/Guardian)
                         - IsPrimaryContact (BIT)
                         - IsBillingResponsible (BIT)
```

### 3.3. Business Justification
* Allows parents to view profiles for multiple children using a single set of login credentials.
* Resolves situations where different family members manage communications vs. payment obligations.

### 3.4. Database Impact
* Creation of `StudentGuardians` join table with composite keys.
* Indexes on `StudentGuardians.GuardianId` to speed up dashboard queries for parents with multiple children.
* Business checks enforce that only one guardian is flagged as `IsBillingResponsible = 1` per student.

### 3.5. API Impact
* Parent portal queries filter students using the logged-in user's `GuardianId` to return child listings.

### 3.6. Future Scalability Impact
Enables easy integration of automated authorization systems (such as RFID pick-up gates validating matching guardian profiles).

## 4. Attendance Status Design

### 4.1. Decision Area: Daily Attendance Codes
Simplifying attendance options for teachers while retaining required board metrics.

### 4.2. Recommended Approach
Define four primary attendance codes for the MVP:
* **Present (P):** Student is physically in class.
* **Absent (A):** Student is absent; triggers an automated notification.
* **Leave (L):** Absence pre-approved by the admin/parent; blocks automated absentee notifications.
* **Late (T):** Student arrived late; logs entry time but counts as present for board reporting.

*Decision:* Defer `Half Day` status to Version 1.5. K-12 schools rarely require mid-day attendance checks for the MVP.

### 4.3. Business Justification
Minimizes input fields on mobile browsers, reducing teacher logging times to under 1 minute. It also prevents sending incorrect absence warnings to parents who have already notified the school.

### 4.4. Database Impact
* Constraints on `DailyAttendance.Status` limit values to `P`, `A`, `L`, `T`.
* Columns added: `CheckInTime` (nullable time value for `Late` tracking).

### 4.5. API Impact
* The API triggers WhatsApp alert dispatches only when status is marked `A`.

### 4.6. Future Scalability Impact
Provides clean percentage metrics for student dashboards.

## 5. Fee Concession & Scholarship Model

### 5.1. Decision Area: Discount Architecture
Defining how concessions (scholarships, staff child discounts, sibling discounts) are applied.

### 5.2. Recommended Approach
Concessions are applied as line-item deductions to standard bills.
* **Calculation Types:** Support both Percentage (e.g., 100% Tuition Fee waiver) and Fixed Amount (e.g., ₹2,000 flat discount).
* **Ledger Security:** Standard bills are generated at full price. Approved concessions are recorded in a separate ledger table (`Concessions`) and applied dynamically:
  $$\text{Net Due} = \text{Billed Amount} - \text{Approved Concession} - \text{Payments}$$
* **Approval workflow:** Concessions are requested by the accountant, require an administrative log approval ID, and freeze once the invoice is finalized.

### 5.3. Business Justification
Enforces accounting transparency. It prevents teachers or accountants from manually reducing standard bill rates, minimizing internal financial leakages.

### 5.4. Database Impact
* Creation of `Concessions` table tracking `StudentId`, `FeeLineItemId`, `ConcessionType` (`Percentage` / `Fixed`), `Value`, `ApprovalAdminId`, and `Notes`.

### 5.5. API Impact
* Invoicing endpoints compute discounts dynamically during invoice creation.

### 5.6. Future Scalability Impact
Supports easy reporting updates (e.g., summarizing total scholarship expenses for school trust reviews).

## 6. Transfer Certificate (TC) Workflow

### 6.1. Decision Area: Leaving Process
Process mapping for students leaving the school.

### 6.2. Recommended Approach
Implement an automated validation workflow:
1. **TC Application Request:** Status changed to pending.
2. **Clear Dues Audit Check:** DB query checks that student ledger balance equals zero.
3. **TC Number Allocation:** Generate sequential identifier: `TC/[ActiveYear]/[Sequence]` (e.g., `TC/2026/0014`).
4. **Profile Lockout:** Student status transitions to `Transferred`. Current year class maps are disabled.

### 6.3. Business Justification
Prevents school staff from issuing TCs to families with outstanding fee balances, resolving a major operational pain point for owners.

### 6.4. Database Impact
* Transaction logs store generated TC details: TC Number, Issue Date, Receiver Name, and Dues Cleared flag.
* Unique key constraints enforce TC number uniqueness.

### 6.5. API Impact
* The API blocks TC generation if the database query returns `OutstandingBalance > 0`.

### 6.6. Future Scalability Impact
Allows the system to maintain a verified registry of former students for board reporting.

## 7. Teacher Assignment Model

### 7.1. Decision Area: Subject and Class Mapping
Structure how teachers are assigned to class sections and specific subjects.

### 7.2. Recommended Approach
Implement an explicit assignment mapping table (`TeacherClassSubjectAssignment`):
* Key attributes: `TeacherId` (FK), `ClassSectionId` (FK), `SubjectId` (FK), `AcademicYearId` (FK).

### 7.3. Business Justification
Avoids hardcoding teacher-class mappings. It allows a teacher to teach different subjects to different sections, handles class teachers (homeroom assignments), and supports mid-year teacher replacements.

### 7.4. Database Impact
* Creation of `TeacherClassSubjectAssignment` table with composite index constraints to prevent duplicate primary teachers for the same class-section-subject combination.

### 7.5. API Impact
* Attendance and grading entry APIs check this table to verify the requesting teacher's edit permissions.

### 7.6. Future Scalability Impact
Enables future scheduling and substitute teacher recommendations.

## 8. Notification Tracking Strategy

### 8.1. Decision Area: Alert Delivery Logs
Tracking notifications (SMS/WhatsApp) dispatched to parents.

### 8.2. Recommended Approach
Implement a dedicated transaction log table (`NotificationLogs`):
* **Status Enum:** `Pending` -> `Sent` -> `Delivered` / `Failed`.
* **Read Status Tracking:** For MVP, only track API delivery statuses. Defer `Read` status tracking (which requires webhook callbacks from WhatsApp for blue ticks) to version 1.5.

### 8.3. Business Justification
Allows the school to verify that parent notifications were sent (e.g. for emergency holiday announcements), reducing operational disputes.

### 8.4. Database Impact
* `NotificationLogs` table stores:
  * `RecipientPhone`
  * `MessageBody` (max 500 characters)
  * `Status` (`VARCHAR(15)`)
  * `RetryCount` (default 0)
  * `ErrorMessage` (nullable string for carrier errors)

### 8.5. API Impact
* Background worker updates log row states on API responses.

### 8.6. Future Scalability Impact
Supports easy reporting of monthly message credit usage.

## 9. Homework Scope Validation

### 9.1. Decision Area: Homework MVP Scope
* **Option A:** Teacher publishes homework cards only (Parent/Student view-only feed).
* **Option B:** Teacher publishes homework, and students upload digital submissions.

### 9.2. Recommended Approach
**Option A (Publish and View Only).** 

### 9.3. Business Justification
In Indian K-12 private schools, homework is written and graded in physical notebooks. Managing digital submissions in the MVP requires:
* Uploading documents from students, consuming storage.
* Complex file screening pipelines.
* Detailed teacher grading dashboards.

Option A meets the school's primary need: providing parent visibility to reduce WhatsApp communication noise. It saves up to 3 weeks of development time for a solo developer.

### 9.4. Database Impact
* Simple `Homework` table schema (Title, Description, Subject, Class, Section, AttachmentLink). No submission tables are created.

### 9.5. API Impact
* Simplified endpoints: Teachers post; Parents read. No submission or feedback pipelines are required.

### 9.6. Future Scalability Impact
A student submission table can be added in version 2 without changing the base tables.

## 10. Student Status History Audit Strategy

### 10.1. Decision Area: Student Status Change Audits
Designing a system to track historical status changes of a student across their lifecycle to support academic transcript verification and school compliance reports.

### 10.2. Recommended Approach
Create a dedicated historical audit table (`StudentStatusHistory`) tracking each transition:
* **Write Mechanism:** Entries are created automatically via a transactional repository interceptor in the API or via database triggers when a student's status changes.
* **Transition Logic Rules:**
  * **Allowed transitions:**
    * `Applied` ──► `Admitted`
    * `Admitted` ──► `Active`
    * `Active` ──► `Promoted` / `Repeating` / `Transferred` / `Dropped` / `Graduated`
    * `Promoted` / `Repeating` ──► `Active` (triggered during academic rollover)
    * `Transferred` / `Dropped` / `Graduated` ──► `Archived`
  * **Prohibited transitions:**
    * `Archived` ──► Any other status (archived state is terminal).
    * `Graduated` ──► `Active` / `Admitted` / `Applied` (prevents altering closed graduation records).
    * `Transferred` ──► `Active` (requires re-application and new Admission record to prevent billing leaks).
* **Retention Policy:** Records are immutable and retained indefinitely. They are excluded from standard database purges to comply with board audit requirements.

### 10.3. Business Justification
Prevents administrative fraud (e.g. manually restoring a dropped student to active status to hide unpaid fees) and provides clear data to trace student histories during inspections.

### 10.4. Database Impact
* Creation of `StudentStatusHistory` table:
  * `StudentStatusHistoryId` (UNIQUEIDENTIFIER PK)
  * `TenantId` (UNIQUEIDENTIFIER FK)
  * `StudentId` (UNIQUEIDENTIFIER FK)
  * `OldStatus` (`VARCHAR(20)`)
  * `NewStatus` (`VARCHAR(20)`)
  * `ChangeReason` (`VARCHAR(250)`)
  * `ChangedByUserId` (UNIQUEIDENTIFIER FK)
  * `ChangedOn` (DATETIME2 UTC)
  * `AcademicYearId` (UNIQUEIDENTIFIER FK)
* Non-clustered composite index on `(StudentId, ChangedOn)` to optimize transcript query speeds.

### 10.5. API Impact
* Modification endpoints execute status changes and audit insertions within a single database transaction context.

### 10.6. Future Scalability Impact
Separates active data checks from audit searches, keeping the primary `Students` table small and fast.

## 11. Tenant Settings & Configuration Strategy

### 11.1. Decision Area: Centralized Tenant Configuration
Creating a scalar configurations schema to parameterize school operations and avoid hardcoding values.

### 11.2. Recommended Approach
Adopt a **Singleton Tenant Configuration Row** database structure using a dedicated `TenantSettings` table.
* **Scope Bounds (What belongs in Settings vs. Separate Tables):**
  * *TenantSettings:* Metadata (School Code, Name, Logo Path), timezone and formatting preferences (Time Zone, Date Format), system rules (Attendance Cutoff Time, Receipt Prefix, TC Prefix), and active year indicators (`ActiveAcademicYearId`).
  * *Separate Integration/Secrets Tables:* SMS provider keys, email gateways SMTP passwords, and Razorpay API secret key tokens. These must be stored in a separate table (`TenantSecrets`) with column-level encryption or resolved directly from server-side environment variables to ensure key isolation.
* **Application Load Mechanism:** Configurations are loaded on application start and cached in the .NET API application memory or local Redis cache with an absolute expiration of 24 hours. Cache clearing is triggered via invalidation tokens whenever the settings modification endpoint is called.

### 11.3. Business Justification
Enables a single deployment instance to run different rules for different schools (e.g., School A has a cutoff time of 08:30 AM and uses `SCH-A` as its receipt prefix, while School B has a cutoff of 09:00 AM and uses `SCH-B`). This is critical for scaling a multi-tenant SaaS.

### 11.4. Database Impact
* Creation of `TenantSettings` table:
  * `TenantId` (UNIQUEIDENTIFIER PK/FK)
  * `ActiveAcademicYearId` (UNIQUEIDENTIFIER FK)
  * `SchoolCode` (`VARCHAR(20)` Unique index)
  * `SchoolName` (`VARCHAR(150)`)
  * `LogoPath` (`VARCHAR(500)`)
  * `AttendanceCutoffTime` (`TIME`)
  * `ReceiptPrefix` (`VARCHAR(10)`)
  * `TCPrefix` (`VARCHAR(10)`)
  * `TimeZone` (`VARCHAR(50)` Default 'India Standard Time')
  * `DateFormat` (`VARCHAR(20)` Default 'dd/MM/yyyy')
  * `CreatedOn` (`DATETIME2`)
  * `ModifiedOn` (`DATETIME2`)

### 11.5. API Impact
* Background services check the cached `AttendanceCutoffTime` before accepting attendance submissions.
* Auto-generation endpoints query the cached prefixes during invoice and TC registration.

### 11.6. Future Scalability Impact
New settings variables (e.g. multi-currency indicators, grading schemas) can be appended as columns to `TenantSettings` without disrupting base query designs.
