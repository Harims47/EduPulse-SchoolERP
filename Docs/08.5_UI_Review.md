# UI/UX Screen Inventory Audit & Review
**Project Name:** EduPulse AI  
**Role Perspective:** Senior Product Designer & School ERP UX Expert  
**UI Framework:** React with Bootstrap 5 (Mobile-Responsive layouts)  
**Status:** Audit Completed - Lean MVP Guidelines Confirmed  

---

## 1. Executive Summary

This audit reviews the proposed user interface screens in `08_UI_Screen_Inventory.md`. While the screen list covers standard student and billing operations, several critical administrative workflows, data portals, and navigation paths are missing from the user journey. 

To ensure the MVP has no dead-ends and is fully usable by non-technical school staff, this review identifies the missing screens and details their layout structures, navigation transitions, user role restrictions, and backend API contracts.

### 2.1. Student 360-Degree View Screen (`Student360View`)
* **Purpose:** A centralized read-only dashboard for administrators, teachers, and parents to view a student's history (academics, attendance rates, billing details, and active files) without switching modules.
* **User Roles:** `SchoolAdmin`, `Accountant`, `Teacher`, `Parent`
* **Navigation Flow:** 
  * Admin/Teacher: Student Directory ──► Click Student Name ──► `/students/view/{studentId}`
  * Parent: Parent Dashboard ──► Click Child Profile ──► `/students/view/{studentId}`
* **UX Component Structure:**
  * **Header Banner:** Student Photo, Name, Admission Number, Class Section, and Status Badge (e.g. `Active` in Green, `Transferred` in Red).
  * **Card Metrics Grid:** Quick-view widgets showing attendance percentage (e.g. `96.4%`), outstanding balance (e.g. `₹15,000`), and latest GPA.
  * **Tabbed Navigation:**
    * **Profile Tab:** Guardian details, address, social category, and Aadhaar index.
    * **Academics Tab:** Exam score cards and historical report card PDF downloads.
    * **Billing Tab:** Paginated list of student invoices, payment receipts, and concessions.
    * **Attendance Tab:** Monthly calendar grid highlighting absences (Red), leaves (Yellow), and presence (Green).
* **API Dependencies:** `GET /api/students/{id}`, `GET /api/invoices`, `GET /api/attendance/student/{id}`

### 2.2. Year-End Rollover Screen (`AcademicYearRolloverBoard`)
* **Purpose:** A single-page execution board to validate rollover prerequisites, view error logs, and trigger batch student promotions (replacing complex step-by-step wizard UI).
* **User Roles:** `SchoolAdmin`
* **Navigation Flow:** Sidebar: Settings ──► Academic Year Directory ──► Click "Run Rollover" ──► `/settings/rollover`
* **UX Component Structure:**
  * **Section A: Preconditions Check:** A card containing a "Validate Rollover Preconditions" button. Tapping this triggers validation checks and populates error listings if failures exist.
  * **Section B: Validation Results Table:** Lists active warnings and blockers (e.g., draft exams, unsubmitted attendances, or uncollected fee dues list). If blockers exist, the Execution card below remains locked.
  * **Section C: Rollover Execution Panel:** Locked card featuring a target year selector dropdown, a summary check indicator of students to be promoted vs. repeated, and a "Run Batch Rollover" final action button.
* **API Dependencies:** `POST /api/academic-years/rollover/validate`, `POST /api/academic-years/rollover/execute`

### 2.3. Transfer Certificate Desk (`TransferCertificateManager`)
* **Purpose:** Centralized console to manage student exits, verify clear dues status, generate TC numbers, and compile official exit certificates.
* **User Roles:** `SchoolAdmin`, `Accountant`
* **Navigation Flow:** Sidebar: Students ──► TC Manager (`/students/tc-manager`)
* **UX Component Structure:**
  * **TC Requests Grid:** Lists students with pending exit applications.
  * **Clear Dues Status indicator:** Color-coded label showing `Outstanding Balance = ₹0` (Green) or `Dues Unpaid = ₹2,500` (Red) to enforce financial clearance rules.
  * **Voucher Details Panel:** Quick view of the student's final billing ledger.
  * **Generate TC Trigger:** A confirmation button that locks the student profile, registers the exit inside history logs, and opens a printable Transfer Certificate template.
* **API Dependencies:** `GET /api/reports/fee-defaulters`, `POST /api/students/{id}/tc/issue`

### 3.4. Receipt History & Cancellation Desk (`ReceiptHistoryDesk`)
* **Purpose:** Audit history of cashier payment logs, supporting receipt printing, search queries, and cancellations for billing corrections.
* **User Roles:** `Accountant`, `SchoolAdmin`
* **Navigation Flow:** Sidebar: Fees ──► Payment Receipts (`/fees/receipts`)
* **UX Component Structure:**
  * **Payment Log Table:** Searchable list of issued vouchers.
  * **Modal: Void/Cancel Receipt:** Input modal requiring a cancellation reason (e.g., cheque bounce details) to log auditing trails before executing reversals.
* **Grid Columns:** `ReceiptNo`, `InvoiceNo`, `StudentName`, `AmountPaid`, `PaymentDate`, `PaymentMethod`, `Status` (Active/Cancelled), `Actions`.
* **API Dependencies:** `POST /api/receipts/{id}/cancel`

## 4. Mobile Usability Guidelines

Because parents and teachers primarily access the platform on mobile browsers in India, the Bootstrap UI must enforce these responsive designs:
1. **Attendance Button Matrix:** For the `DailyAttendanceSheet` screen, avoid complex swipe gestures (swipe-to-mark). Use simple, large, touch-friendly button sets (`P`, `A`, `L`, `T`) in a grid layout. This is faster to build, requires no external gesture libraries, and handles all four statuses cleanly.
2. **Fixed Bottom Actions on Mobile:** Operational forms (such as grade entries or cashier desks) must place saving buttons inside a sticky footer at the bottom of the viewport to prevent users from scrolling endlessly to find action buttons.
3. **No Heavy Modal Forms on Parent Dashboard:** Modal views do not scale well on small screens. Parent alerts or invoice cards should expand inline using collapsing sections instead of modal windows.
