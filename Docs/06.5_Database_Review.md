# SQL Server Database Design Audit & Review
**Project Name:** EduPulse AI  
**Role Perspective:** Senior SQL Server Database Architect & School ERP Domain Expert  
**Status:** Architecture Audit Completed  

---

## 1. Executive Summary

This audit reviews the proposed database design schema for the MVP of **EduPulse AI** (`06_Database_Design.md`). While the schema provides a functional base for multi-tenancy and core school operations, a critical review from a Database Architect and Indian School ERP Domain Expert reveals several structural gaps, data integrity risks, and auditing limitations that would compromise standard accounting rules, CBSE board reporting, and administrative operations.

This document details the identified vulnerabilities and proposes concrete database modifications to make the schema implementation-ready.

---

## 2. Review of Core Table Structures

### 2.1. The Student Table
* **Vulnerability:** Lacks standard audit metadata (`CreatedBy`, `CreatedOn`, `ModifiedBy`, `ModifiedOn`) and critical demographic indicators mandatory for CBSE/State Board registries. Sibling relationships are represented only implicitly through the guardian junction, which complicates sibling fee concession rules.
* **Missing Indian K-12 Fields:** 
  * `AadhaarNo` (Mandatory for government compliance, requires encryption).
  * `SocialCategory` (Gen / OBC / SC / ST - required for state scholarship allocation and statistical reports).
  * `MotherTongue` (Required for board registration).
* **CTO Recommendation:** Add audit columns, explicit demographic columns, and an encrypted `AadhaarNo` field.

### 2.2. The Staff Table
* **Vulnerability:** Lacks a designation or departmental model. A staff member is simply linked to a user profile, but the database cannot query who is a "PGT (Post Graduate Teacher)", "TGT (Trained Graduate Teacher)", "Accountant", or "Admin Assistant" without parsing system roles, which are security constructs, not organizational models.
* **CTO Recommendation:** Introduce a `Designation` field and explicit department assignments.

### 2.3. The Fee & Ledger Tables
* **Vulnerability (Critical Accounting Failure):** The schema maps payments to the entire invoice (`FeeReceipts.InvoiceId`) but **lacks a line-item allocation table**. If a parent pays ₹5,000 against a ₹10,000 invoice (comprising Tuition Fee, Transport Fee, and Lab Fee), the database cannot track *which specific line items were cleared* and *which remain outstanding*.
* **CTO Recommendation:** Introduce a `FeeReceiptAllocations` junction table. This is vital to allow the Accountant to run reports such as *"Total Tuition Fees Collected vs. Total Transport Fees Collected."*

### 2.4. Teacher Assignment Model
* **Vulnerability:** The schema maps subject teaching assignments but lacks a mechanism to designate the **Class Teacher (Homeroom Teacher)** for a class section. In Indian schools, the Class Teacher is a critical legal and operational role responsible for signing off on report cards and finalizing daily attendance.
* **CTO Recommendation:** Add a `IsClassTeacher` column to the `ClassSections` table or create an explicit mapping in the assignment model.

### 2.5. Guardian Model
* **Vulnerability:** Lacks structural validation rules. Sibling linkage depends on matching phone numbers or matching guardian records, which can lead to orphaned accounts if typographical errors occur during entry.
* **CTO Recommendation:** Standardize address fields and configure indexes to validate unique primary parent keys.

### 3.1. Missing Domain Master Tables

#### Table Name: `FeeReceiptAllocations` [NEW]
* **Purpose:** Distributes a single payment receipt amount across the individual billed line items of an invoice, supporting partial payment accounting.
* **Database Design:**
  * `AllocationId` (`UNIQUEIDENTIFIER` PK, `DEFAULT NEWSEQUENTIALID()`)
  * `TenantId` (`UNIQUEIDENTIFIER` FK to `Tenants`)
  * `ReceiptId` (`UNIQUEIDENTIFIER` FK to `FeeReceipts`)
  * `InvoiceDetailId` (`UNIQUEIDENTIFIER` FK to `StudentInvoiceDetails`)
  * `AmountAllocated` (`DECIMAL(18,2)`)
  * `CreatedOn` (`DATETIME2`, `DEFAULT SYSUTCDATETIME()`)
* **Unique Constraints:** `UQ_FeeReceiptAllocations_ReceiptDetail` on `(TenantId, ReceiptId, InvoiceDetailId)`
* **Check Constraints:** `CK_FeeReceiptAllocations_Amount`: `AmountAllocated > 0`

#### Table Name: `StudentDocuments` [NEW]
* **Purpose:** Normalizes document attachments (Aadhaar scan, Transfer Certificate, Birth Certificate) away from the main `Students` table to allow flexible uploads.
* **Database Design:**
  * `DocumentId` (`UNIQUEIDENTIFIER` PK, `DEFAULT NEWSEQUENTIALID()`)
  * `TenantId` (`UNIQUEIDENTIFIER` FK to `Tenants`)
  * `StudentId` (`UNIQUEIDENTIFIER` FK to `Students`)
  * `DocumentType` (`VARCHAR(30)`) -- e.g. 'Aadhaar', 'TC', 'BirthCertificate'
  * `FileName` (`NVARCHAR(250)`)
  * `FilePathKey` (`VARCHAR(500)`)
  * `UploadedOn` (`DATETIME2`, `DEFAULT SYSUTCDATETIME()`)
  * `UploadedByUserId` (`UNIQUEIDENTIFIER` FK to `Users`)
* **Check Constraints:** `CK_StudentDocuments_Type`: `DocumentType IN ('Aadhaar', 'TC', 'BirthCertificate', 'Other')`

### 3.2. Mandatory Column Additions to Existing Tables

#### Table: `Students` [MODIFY]
* **Recommended Changes:**
  * Add `AadhaarNo` (`VARCHAR(100)` Nullable) - Encrypted.
  * Add `SocialCategory` (`VARCHAR(10)` Nullable) - Check constraints: `GEN`, `OBC`, `SC`, `ST`.
  * Add `MotherTongue` (`VARCHAR(50)` Nullable) - Default 'Hindi' or local languages.
  * Add standard tracking columns: `CreatedOn` (`DATETIME2`), `CreatedByUserId` (`UNIQUEIDENTIFIER`), `ModifiedOn` (`DATETIME2`), `ModifiedByUserId` (`UNIQUEIDENTIFIER`).
* **Database Impact:** Requires column-level encryption keys for `AadhaarNo`. Enlarges row size slightly.
* **Development Impact:** The C# DTOs and Dapper parameters must explicitly pass these audit parameters.

#### Table: `ClassSections` [MODIFY]
* **Recommended Changes:**
  * Add `ClassTeacherId` (`UNIQUEIDENTIFIER` Nullable) - FK referencing `Staff(StaffId)`.
* **Database Impact:** Requires validation that the assigned staff member is active.
* **Development Impact:** Simple dropdown on the React class settings page to assign the homeroom teacher.

#### Table: `Staff` [MODIFY]
* **Recommended Changes:**
  * Add `Designation` (`VARCHAR(50)` Nullable) - e.g., 'TGT Science', 'PGT Physics', 'Accountant'.
  * Add standard tracking columns: `CreatedOn` (`DATETIME2`), `CreatedByUserId` (`UNIQUEIDENTIFIER`), `ModifiedOn` (`DATETIME2`), `ModifiedByUserId` (`UNIQUEIDENTIFIER`).
* **Database Impact:** None. Simple metadata additions.

## 4. Key Architectural & Scalability Risks

### 4.1. Audit Trail Bloat
* **Risk:** The general `AuditLogs` table stores JSON payloads in an `NVARCHAR(MAX)` column. With active use across 5 schools, this table will accumulate millions of logs, leading to database file bloat and slower backups.
* **Mitigation:**
  1. Restrict JSON logs to critical transactions (fee allocations, concessions, grade modifications).
  2. Implement SQL Server partition schemes on `AuditLogs` using the `Timestamp` column to archive logs older than 1 year to secondary filegroups.

### 4.2. soft Delete Reference Anomalies
* **Risk:** Soft-deleting a student (`IsDeleted = 1`) can cause reporting errors if dependent records (like `StudentClassHistory` or `DailyAttendance`) do not check this flag.
* **Mitigation:** Database views (e.g., `vw_ActiveStudents`) should be used for common queries, ensuring soft-deleted rows are filtered out by default.

### 4.3. Currency and Scale Precision
* **Risk:** Using `DECIMAL(18,2)` is standard, but rounding issues can occur during partial payments or concessions.
* **Mitigation:** Implement strict rounding policies at the API level before executing Dapper commands, ensuring calculations match double-entry rules.
